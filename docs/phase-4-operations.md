# Phase 4 — 운영 및 관리자 기능

서비스가 출시 직전에 가까워지면 **운영자 도구**가 필요합니다.
이 단계의 목표는 외부 노출 없이 내부 운영자가 즉시 대응할 수 있게 하는 것.

---

## 1. 관리자 전용 화면

### 1-1. 접근 제어
- `profiles.role` 컬럼 추가(`'user' | 'admin'`).
- `/admin/*` 라우트는 미들웨어에서 role 검사 → 비-관리자는 404.
- 추가로 RLS 정책에서 `auth.jwt() ->> 'role' = 'admin'`을 확인.

### 1-2. 기능
| 화면 | 경로 | 기능 |
| --- | --- | --- |
| 스테이지 관리 | `/admin/stages` | 보드 구성/목표/시간 수정, 즉시 배포(soft publish) |
| 유저 관리 | `/admin/users` | 닉네임/접속 이력, **차단(`is_blocked = true`)** 토글 |
| 신고 처리 | `/admin/reports` | `reports` 테이블 큐, 상태 변경(open → resolved) |
| 가챠 모니터 | `/admin/gacha` | 풀별 사용자 분포 / 평균 비용 / 이상 패턴 알림 |

### 1-3. 감사 로그
- 모든 관리자 액션은 `admin_audit_log` 테이블에 기록.
- 최소 기록: `admin_id`, `action`, `target_table`, `target_id`, `payload jsonb`, `created_at`.
- 재현 가능한 변경(되돌리기)을 위해 변경 전후 스냅샷 저장.

> 본 가이드 폴더 구성에서는 `/admin` 라우트와 컴포넌트 골격만 둡니다.
> 실제 테이블/UI는 출시 후 운영팀과 함께 완성합니다.

---

## 2. 고객센터 연동

### 2-1. 앱 내 문의하기
- `src/app/my/page.tsx`의 "문의하기" 카드 → 외부 채널 링크.
  - 1순위: **카카오 채널** (한국 사용자 친화).
  - 2순위: 이메일 `mailto:` 링크 (앱스토어 심사 시 메타데이터에도 등록).
- 미니앱 환경에서는 `window.open` 대신 Capacitor의 `Browser.open()` 사용해 인-앱 브라우저로 표시.

### 2-2. FAQ 섹션
- 정적 마크다운 → 빌드 시 변환해 노출(검색 엔진 노출이 필요 없음).
- 카테고리: 계정/결제/가챠/플레이/제재/탈퇴.
- "이 답변이 도움이 되었나요?" 피드백을 누적해 자주 보는 항목 상위 노출.

### 2-3. 운영 워크플로
1. 사용자가 문의/신고 → 카카오 채널 또는 in-app `/admin/reports` 큐.
2. 운영자가 24시간 내 1차 응답.
3. 7일 이상 미해결 건은 자동 알림(슬랙 webhook).
